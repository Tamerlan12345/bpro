# Предложения по улучшению UI/UX для сервиса BizMap AI

## 1. Редизайн и улучшение визуала

### Эстетика и Цветовая Палитра
Текущая палитра использует базовые оттенки синего, серого и зеленого. Предлагается перейти на более глубокие, профессиональные цвета и использовать систему дизайн-токенов Tailwind-стиля.

**Новая концепция цветов (Токены):**
*   **Primary (Основной):** Переход от `#2a6fdb` к более современному `#3b82f6` (или `#2563eb` для контраста).
*   **Primary Hover:** `#1d4ed8`
*   **Surface/Background (Фон):** Текущий `#f8f9fa` хорош, но можно сделать его чуть более холодным `#f8fafc`.
*   **Surface Secondary (Карточки):** Чистый белый `#ffffff` с мягкими тенями.
*   **Text (Текст):** Смягчить черный, использовать `#1e293b` для заголовков и `#475569` для основного текста.
*   **Danger (Ошибка/Удаление):** `#ef4444`, hover: `#dc2626`.
*   **Success (Успех):** `#10b981`.

### Типографика
Шрифт `Inter` является отличным выбором. Необходимо лишь немного настроить иерархию (line-height и letter-spacing):
*   Увеличить `line-height` для абзацев (до `1.6` или `1.7`).
*   Слегка уменьшить `letter-spacing` для крупных заголовков (например, `-0.02em`), чтобы они выглядели плотнее и современнее.

### Макет и Композиция
*   **Асимметрия колонок:** Текущее разделение `1.5fr 1fr` в `.main-content` работает хорошо, но при больших экранах (свыше 1600px) контент может слишком сильно растягиваться. Рекомендуется ограничить максимальную ширину колонок или использовать центрирование внутри них.
*   **Отступы:** В CSS используются токены `var(--space-md)`, но местами встречаются жестко заданные значения (например, `padding: 25px;` в `.modal-content`). Переход исключительно на переменные улучшит консистентность.

## 2. Улучшение отображения и работы кнопок

### Дизайн кнопок (Кнопки действий, проверок)
*   **Состояния:** Сейчас кнопки имеют состояния обычное, наведение и `disabled`. Необходимо добавить явное состояние `active` (нажатие) с небольшим уменьшением (`transform: scale(0.98)`) для тактильной отдачи.
*   **Фокус (Accessibility):** Кнопки и инпуты должны иметь более явный `outline` или `box-shadow` при фокусе (например, `box-shadow: 0 0 0 3px rgba(59, 130, 246, 0.5)`).
*   **Иконки:** Использовать больше векторных иконок (SVG) внутри кнопок (например, в кнопках загрузки, записи, аудита) для лучшего визуального якоря.

### Обратная связь
*   **Спиннеры:** Текущая реализация (класс `.spinner`) хороша, но можно добавить анимацию появления/затухания самого текста кнопки при замене на спиннер, чтобы избежать резких скачков ширины кнопки.
*   **Тосты (Уведомления):** Функция `showNotification` работает, но анимация скрытия жестко задана (`fadeOut 0.5s 4.5s forwards`). Желательно добавить кнопку закрытия на самих уведомлениях и позволить им "складываться" (stacking) более плавно.
*   **Ошибки в модальных окнах:** Если сохранение (например, транскрибации) падает, ошибка должна отображаться внутри модального окна, а не только как тост, чтобы пользователь понимал контекст.

## 3. Отчет об ошибках (Баги и узкие места)

### Функциональные баги

**1. Использование встроенных алертов браузера**
*   **Краткое описание:** При попытке скачать диаграмму без предварительной генерации вызывается встроенный `alert()`.
*   **Шаги для воспроизведения:** Открыть страницу чата, не вводить текст и не генерировать схему. Нажать кнопку "Скачать SVG" или "Скачать для MS Visio".
*   **Ожидаемый результат:** Красивое всплывающее уведомление через систему `showNotification` ("Сначала сгенерируйте схему.").
*   **Фактический результат:** Появляется системный браузерный `alert("Сначала сгенерируйте схему.");` (строка 340 в `script.js`).

**2. Конфликты z-index и отображение модальных окон**
*   **Краткое описание:** Использование высоких жестких значений `z-index` может привести к перекрытию элементов в будущем.
*   **Шаги для воспроизведения:** Открыть несколько слоев интерфейса (например, модальное окно поверх уведомлений, или контекстное меню Cytoscape).
*   **Ожидаемый результат:** Элементы должны наслаиваться согласно строгой системе z-index.
*   **Фактический результат:** В CSS вразнобой используются значения `10`, `100`, `1050` и даже `10000` (`#process-info-modal` инлайн). Это потенциальное узкое место для поддержки.

### Визуальные баги

**1. Жестко заданные !important и размеры в премиум-модалке**
*   **Краткое описание:** Модальное окно премиум-версии (`.premium-modal`) использует `max-width: 650px !important;` и `padding: 0 !important;`.
*   **Шаги для воспроизведения:** Открыть `.premium-modal` на мобильном устройстве (экран < 600px).
*   **Ожидаемый результат:** Модальное окно адаптируется под экран, сохраняя внутренние отступы.
*   **Фактический результат:** Использование `!important` усложняет переопределение стилей через медиа-запросы, что может привести к обрезанию контента или прилипанию текста к краям на узких экранах.

**2. Дублирование стилей кнопок закрытия (.close-btn)**
*   **Краткое описание:** В `style.css` класс `.close-btn` определен дважды с разными стилями.
*   **Шаги для воспроизведения:** Просмотреть файл `style.css`.
*   **Ожидаемый результат:** Единый, переиспользуемый класс для всех кнопок закрытия.
*   **Фактический результат:** На строках 696 и 989 определены разные свойства, что может вызывать "прыжки" кнопки или некорректное отображение в зависимости от порядка применения стилей.

**3. Инлайн стили для скрытия элементов (`display: none;`)**
*   **Краткое описание:** Множество элементов скрыты через инлайн-стили в HTML, что делает их жестко зависимыми от JS.
*   **Шаги для воспроизведения:** Попытаться переопределить видимость элемента через CSS класс (например, для мобильных устройств).
*   **Ожидаемый результат:** Управление видимостью через CSS-классы (например, `.hidden`, `.d-none`), которые можно менять в JS.
*   **Фактический результат:** Инлайн `style="display: none;"` перебивает любые CSS правила (кроме `!important`), усложняя адаптивную верстку. (Учитывая память проекта, мы не меняем эти атрибуты, но это является узким местом архитектуры UI).
